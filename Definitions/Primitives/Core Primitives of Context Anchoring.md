> **Context Anchoring is a prompt-native computation model built on three core primitives: Anchor, Gate, and Loop. Together, they provide explicit state, bounded logic, and controlled transitions within and between Prompt Compute Runtimes.**

This document defines the stable core structure and vocabulary of the framework.

---

## The Big Idea

Traditional prompting often treats each request as an isolated instruction.

Context Anchoring treats the active prompt and context as a temporary Compute Runtime.

That runtime is built from three primitives:

```text
ANCHOR(S) -> GATE(S) -> LOOP (optional) -> UPDATED OR DERIVED ANCHOR(S)
```

Everything else, including intent, constraints, auditing, confirmation, compression, skills, orchestration, and state transfer, is implemented as content or behavior associated with these primitives.

The framework is:

- Modular
- Stateful within an active runtime
- Transferable across runtimes through explicit state handoffs
- Testable
- Extensible
- Stability-oriented

Context Anchoring does not require fine-tuning or permanent changes to the model. It initializes task-specific operational state at inference time.

---

## Runtime and External State

The three primitives become active when they are present in the current Prompt Compute Runtime.

External systems may store, route, or retrieve their serialized state. Examples include:

- Markdown files
- JSON or YAML artifacts
- PostgreSQL records
- Vector stores
- Orchestrator state
- Agent handoff messages

External storage does not replace Anchors, Gates, or Loops. It preserves information that a later runtime may retrieve, validate, and load as active Anchor state.

The distinction is:

```text
Stored Data
    -> Retrieved and validated
    -> Loaded as an Anchor
    -> Processed by Gates
    -> Controlled by optional Loops
```

No hidden memory should be assumed. If state matters to a runtime, it must be present in the active context or made available through an explicit retrieval mechanism.

## Primitive Hierarchy

The primitives have distinct responsibilities:

| Primitive | Definition | Analogy | Role |
|---|---|---|---|
| **Anchor** | Highest-ranking named state object | Register or scoped variable | Defines stable, dynamic, or transferable state |
| **Gate** | Named and bounded logic unit | Function or procedure | Reads, transforms, validates, or updates Anchor state |
| **Loop** | Optional control structure containing a Test and Confirmation | Conditional control loop | Controls continuation, retry, rerouting, escalation, or termination |

The Anchor is highest-ranking because the Gate and Loop depend on the state, scope, and authority it defines.

---

## 1. Anchor: Named State

An Anchor is a named and scoped state object that preserves essential context across the steps of a task.

Anchors define:

- Objective
- Scope
- Role and authority
- Current state
- Shared knowledge
- Operational parameters
- Output contracts
- State locations

Anchors may be:

- **Static:** Objective, domain, or rule
- **Dynamic:** Score, progress, or workflow status
- **Derived:** Validated output generated by a Gate
- **Externalized:** Serialized Anchor state stored outside the session
- **Transferred:** Externalized state loaded into another runtime

Example:

```text
A0 - Objective: Teach third-grade weather vocabulary
A1 - Difficulty: Easy
A2 - Domain: Weather terms only
A3 - Progress: {correct: 2, incorrect: 1}
```

Anchors are expected to remain authoritative within their declared scope until explicitly updated, deactivated, or superseded by a higher-priority instruction.

---

## 2. Gate: Bounded Logic

A Gate reads one or more Anchors, performs a defined operation, and produces output or updated Anchor state.

Gates may:

- Generate output
- Retrieve data
- Transform formats
- Evaluate correctness
- Validate a contract
- Call a skill or tool
- Create a derived Anchor
- Update a permitted dynamic Anchor

Formal shapes include:

```text
GATE(INPUT ANCHORS) -> OUTPUT
```

```text
GATE(A_IN) -> A_OUT
```

A Gate does not become deterministic because its inputs are named. Its constraints and tests are intended to improve repeatability within a measured range.

---

## 3. Loop: Tested Control

A Loop is an optional control structure that evaluates a Gate result and determines the next transition.

A Loop contains two essential parts:

1. **Test:** Evaluate output or Anchor state against a defined condition.
2. **Confirmation:** Record whether the condition was satisfied and provide the result used for control flow.

Based on the Confirmation, the Loop may:

- Continue
- Reapply a Gate
- Request missing state
- Reroute execution
- Escalate
- Stop successfully
- Stop with failure

Every repeating Loop requires a stop condition, retry limit, or escalation boundary.

Not every Gate requires a Loop.

---

## Canonical Naming

The canonical notation is:

```text
A1       Anchor 1
A1.G01   Gate 01 associated with Anchor A1
A1.G02   Gate 02 associated with Anchor A1
A1.L01   Loop 01 associated with Anchor A1
```

This convention makes the hierarchy visible and prevents Gates from being confused with Anchor identifiers.

## Cross-Cutting Behaviors

These are not additional primitives. They are content or behaviors layered onto Anchors, Gates, and Loops.

| Behavior | Responsibility |
|---|---|
| **Intent** | Declares the objective, commonly through an Anchor |
| **Constraint** | Defines an operational boundary through Anchor state or Gate rules |
| **Audit or Test** | Evaluates Gate output or Anchor state |
| **Confirmation** | Records the Test result and supporting evidence |
| **Compression** | Reduces validated state while preserving required meaning |
| **State Transfer** | Serializes, stores, retrieves, and reloads Anchor state |
| **Skill Loading** | Makes a bounded capability or procedure available to a runtime |
| **Orchestration** | Selects runtimes, routes state, and schedules execution |

An Audit or Test checks a result. A Confirmation records the outcome. A Loop uses that outcome to control the next transition.

## End-to-End Example

### Step 1: Define Anchors

```text
A0 - Objective: Weather vocabulary
A1 - Difficulty: Easy
A2 - Domain: Weather terms
A3 - Score: 0
```

### Step 2: Run a Gate

```text
A0.G01 - Read A0 through A2 and produce one vocabulary puzzle.
```

### Step 3: Run an Evaluation Gate

```text
A3.G01 - Validate the learner's answer and return the proposed score update.
```

### Step 4: Optionally Run a Loop

```text
A3.L01

TEST:
Confirm that the answer evaluation follows A2 and that the score update is valid.

CONFIRMATION:
Return PASS or FAIL with the current and proposed scores.

ON PASS:
Update A3 and continue.

ON FAILURE:
Reapply A3.G01 once without changing previously validated Anchors.
```

The Anchors preserve state, the Gates perform the work, and the Loop controls the transition.

## Runtime Boundaries

Context Anchoring operates within the usable context of the active model session.

Anchors, Gates, Loops, task data, tool results, intermediate state, and generated output all consume runtime capacity. A practical stability boundary may be reached before the model's hard context limit.

When a runtime approaches that boundary, the system may:

- Compress validated Anchor state
- Externalize state to a durable artifact
- Divide work into smaller Gates
- Transfer a bounded task to another agent
- Start a new runtime with only the required Anchors

Runtime limit is also a design limit. It is model-dependent and task-dependent, not one universal token number.

---

## Reliability Boundary

Context Anchoring improves structure, consistency, testability, and failure visibility. It does not guarantee deterministic inference or eliminate hallucinations.

Where possible, high-impact Confirmations should use observable evidence such as:

- Schema validation
- Programmatic calculations
- Database constraints
- Tool results
- Source verification
- Separate evaluators
- Human review

The practical goal is a runtime whose state, logic, tests, and failures can be inspected and improved.

---

## Closing Summary

Context Anchoring is built on three primitives:

```text
ANCHOR -> GATE -> LOOP
```

- Anchors define state.
- Gates perform bounded logic.
- Loops test results and control transitions.

External storage, skills, orchestration, audits, and state transfer extend the runtime without becoming additional primitives.
