  > **Context Anchoring is a prompt-native computation model built on three core primitives: Anchor, Gate, and Loop.
  Together, they enable explicit state, structured reasoning, and controlled iteration within an LLM's context window.**

  This document defines the current core structure and vocabulary of the framework.

  ---

  ## The Big Idea

  Traditional prompting treats each turn like a fresh start.
  Context Anchoring treats prompting as computation.

  In this model, a prompt becomes a small reasoning engine built from three primitives:

  ```text
  ANCHOR(S) -> GATE(S) -> LOOP (optional) -> updated ANCHOR(S)```

  Everything else, including auditing, constraints, intent, and compression, is layered on top of these primitives
  rather than treated as a primitive itself.

  This is what makes the approach:

  - Modular
  - Stateful
  - Testable
  - Extensible
  - More stable under structured reuse

  And critically:

  > No fine-tuning.
  > No external database.
  > No hidden memory.
  > Everything lives inside the prompt window.

  ———

  ## The Three Primitives

  | Primitive | Definition | Analogy | Role |
  |---|---|---|---|
  | Anchor | A named, persistent context register | Variable in memory | Holds stable state |
  | Gate | A bounded reasoning step that reads or updates anchors | Function | Performs logic |
  | Loop | Optional repetition until a stop condition is met | While-loop | Enables refinement |

  ———

  ## 1. Anchor: Stable Context

  An Anchor is a named reference point that preserves task context across reasoning steps.

  Anchors define:

  - What must remain true
  - The scope of the task
  - The current state
  - Shared knowledge
  - Operational parameters

  Anchors may be:

  - Static: objective, domain, rules
  - Dynamic: score, progress, confidence
  - Derived: outputs generated by gates

  ### Example

  A0 - Objective: 3rd grade weather vocabulary
  A1 - Difficulty: Easy
  A2 - Domain: Weather terms only
  A3 - Progress: {correct: 2, incorrect: 1}

  Think of Anchors like CPU registers: small, stable, and essential.

  ———

  ## 2. Gate: Bounded Reasoning

  A Gate is a bounded reasoning step. It reads Anchors, performs a transformation, and may write updates back to
  Anchors.

  Gates can:

  - Generate outputs
  - Evaluate correctness
  - Transform formats
  - Summarize state
  - Enforce contracts

  ### Formal Shape

  Gate(Input Anchors) -> Output

  Or, when updating state:

  Gate(A_in) -> A_out

  Gates are the functional units of the system.

  ———

  ## 3. Loop: Controlled Iteration

  A Loop repeats one or more Gates until a stop condition is reached.

  Typical stop conditions include:

  - Objective reached
  - Stable output achieved
  - External stop signal
  - Threshold reached

  Not every workflow requires a Loop. But when refinement, self-correction, or progressive summarization is needed, it
  becomes the mechanism for controlled iteration.

  ———

  ## Cross-Cutting Behaviors

  These are not primitives. They are behaviors layered onto Anchors and Gates.

  | Behavior | Role |
  |---|---|
  | Verification (Audit) | Checks correctness and compliance |
  | Constraints | Define operational boundaries |
  | Intent | Declares the objective, often through A0 |
  | Compression | Summarizes state to fit within the context window |
  | State Transfer | Propagates updated anchor state across steps |

  ———

  ## End-to-End Example

  ### Step 1: Define Anchors

  A0 - Objective: Weather vocabulary
  A1 - Difficulty: Easy
  A2 - Domain: Weather terms
  A3 - Score: 0

  ### Step 2: Run a Gate

  Read A0-A2 and produce a puzzle.

  ### Step 3: Run an Evaluation Gate

  Validate the answer and update A3.

  ### Step 4: Optionally Loop

  Repeat until the teacher stops or the objective is reached.

  The Anchors preserve continuity across every step.

  ———

  ## Runtime Boundaries

  Context Anchoring operates within the limits of the model's available context window. In practice, all Anchors, Gates,
  and operational instructions should remain within a stable portion of that window.

  This provides:

  - Full visibility of state
  - Reduced truncation risk
  - More reproducible behavior

  > Runtime limit is also a stability limit.
  > This is a design constraint, not a hard law.

  ———

  ## Closing Summary

  Context Anchoring is built on three primitives:

  > Anchor -> Gate -> Loop

  Together, they turn prompts into programmable systems:

  - Anchors store state
  - Gates perform reasoning
  - Loops refine behavior

  Everything else is layered on top.
